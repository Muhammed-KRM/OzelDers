OzelDers - UI Performans ve Sorun Analizi Dokumani
Tarih: Nisan 2026
==================================================

MIMARI OZET (Bu dokumani okumadan once bilinmesi gerekenler)
------------------------------------------------------------
Projede 3 UI projesi var:

1. OzelDers.Web  (Blazor Web App - .NET 10)
   - Render modu: InteractiveServerRenderMode(prerender: false)
   - Her UI etkilesimi SignalR uzerinden sunucuya gidip geliyor
   - Auth: ProtectedLocalStorage (sifrelenmis, tarayici)
   - Token handler: AuthTokenHandler -> ProtectedLocalStorage okur -> Bearer header ekler
   - API iletisimi: HttpClient -> OzelDers.API (REST)

2. OzelDers.App  (MAUI Blazor Hybrid - .NET 10)
   - Render modu: BlazorWebView (cihazda local, SignalR YOK)
   - Hedef platformlar: Android (min API 24), iOS (min 15.0), Windows, macOS
   - Auth: Preferences.Default (platform native storage)
   - Token handler: MauiAuthTokenHandler -> Preferences okur -> Bearer header ekler
   - API iletisimi: HttpClient -> OzelDers.API (REST)
   - index.html'de body { background-attachment: scroll !important; } duzeltmesi var

3. OzelDers.SharedUI  (Razor Class Library - RCL)
   - Tum sayfalar ve bilesenlerin bulundugu merkezi proje
   - Her iki host (Web ve App) bu projeyi referans alir
   - ApiServices klasoru: IListingService, IAuthService vb. -> HttpClient ile API cagrisi
   - CSS: variables.css, reset.css, animations.css, utilities.css, app.css
   - JS: interop.js (SweetAlert2 wrapper, scrollToTop)
   - Dis bagimliliklar: SweetAlert2 (CDN), Google Fonts (CDN)

Veri akisi her iki platformda ayni:
  Razor sayfasi -> IXxxService (inject) -> XxxApiService (HttpClient) -> OzelDers.API -> PostgreSQL/ES/Redis

SORUN W-1: background-attachment: fixed - Performans Sorunu
  Nerede: src/OzelDers.SharedUI/wwwroot/css/app.css - body blogu
  Mevcut kod: background-attachment: fixed;
  Ne oluyor: Tarayici her scroll hareketinde tum arka plani yeniden boyuyor (repaint).
             Bu ozellik GPU katmanlamasi ile uyumsuz, her karede CPU hesaplamasi gerektirir.
             Mobil tarayicilarda (Chrome Android, Safari iOS) bu ozellik ya hic desteklenmez
             ya da ciddi performans dusumune yol acar. Scroll sirasinda sayfa titrer veya takilir.
  Arastirma: CSS backdrop-filter ve background-attachment:fixed mobil performans sorunlari
             genis capli olarak raporlanmistir. Chrome DevTools'ta "Scrolling performance issues"
             uyarisi verir. App projesinde index.html'de scroll !important ile duzeltilmis
             ama Web projesinde hala fixed duruyor.
  Etki: Web - Orta/Yuksek | App - Dusuk (zaten duzeltilmis)

SORUN W-2: SignalR Baglanti Kopuklugu - Kullanici Deneyimi
  Nerede: OzelDers.Web - InteractiveServerRenderMode mimarisi
  Ne oluyor: Blazor InteractiveServer modunda tum UI etkilesimleri WebSocket (SignalR) uzerinden
             sunucuya gidip geliyor. Baglanti koptuğunda (ag degisimi, uyku modu, zayif sinyal)
             kullanici "Reconnecting..." ekrani goruyor ve sayfa donuyor.
             GitHub dotnet/aspnetcore issue #46950'de belgelenmis: "Blazor Server has higher
             requirements for network bandwidth and lower latency than MVC or Razor Pages."
             Hacker News'te de yaygin sikayet: "not very good at re-establishing the WebSocket
             connection if it fails, giving a poor impression to the user."
  Etki: Web - Yuksek | App - Yok (SignalR kullanmiyor)

SORUN W-3: Ilk Sayfa Yuklemesi Bos Ekran
  Nerede: OzelDers.Web - App.razor
  Mevcut kod: @rendermode="new InteractiveServerRenderMode(prerender: false)"
  Ne oluyor: prerender: false oldugu icin sunucu onceden HTML render etmiyor.
             Blazor.web.js yuklenmeden sayfa tamamen bos gorunuyor.
             Yavaş baglantilar veya sunucu mesgul oldugunda kullanici bos beyaz ekran goruyor.
             Bu SEO icin de kotu: Google bot sayfayi bos gorebilir.
  Etki: Web - Orta | App - Yok

SORUN W-4: Liste Render'larinda @key Direktifi Eksik
  Nerede: src/OzelDers.SharedUI/Pages/Search.razor, Home.razor, MyListings.razor
  Mevcut kod: @foreach (var listing in result.Items) { <ListingCard Listing="listing" /> }
  Ne oluyor: Blazor liste render ederken hangi elemanlarin degistigini anlamak icin diff algoritması
             kullanir. @key olmadan Blazor her eleman icin pozisyon bazli karsilastirma yapar.
             Liste siralama degistiginde veya eleman eklenip cikarildiginda gereksiz DOM guncelleme
             yapilir. InteractiveServer modunda bu her seferinde SignalR uzerinden gider.
             Microsoft resmi dokumantasyonu: "Use @key to control element and component preservation."
  Etki: Web - Orta | App - Dusuk

SORUN W-5: StateHasChanged Gereksiz Cagrilari
  Nerede: Bircok sayfada isLoading flag'i degistirilirken
  Ne oluyor: isLoading = true; -> StateHasChanged tetiklenir -> SignalR -> DOM update
             isLoading = false; -> StateHasChanged tetiklenir -> SignalR -> DOM update
             Bazi sayfalarda bu zincir gereksiz yere birden fazla kez tetikleniyor.
             Her tetiklenme InteractiveServer modunda bir SignalR round-trip demek.
  Etki: Web - Orta | App - Dusuk

SORUN W-6: SweetAlert2 CDN Bagimliligi
  Nerede: src/OzelDers.Web/Components/App.razor
  Mevcut kod: <script src="https://cdn.jsdelivr.net/npm/sweetalert2@11"></script>
  Ne oluyor: CDN erisimi olmadigi durumlarda (kurumsal ag, CDN kesintisi) SweetAlert2 yuklenmez.
             interop.js'de typeof Swal === 'undefined' kontrolu var, fallback alert() kullaniliyor.
             Ama bu kullanici deneyimini bozuyor. Ayrica CDN'den yukleme ilk sayfa acilisini
             yavaslatir (DNS lookup + TCP + TLS + download).
  Etki: Web - Dusuk/Orta | App - Yuksek (offline/zayif ag)

SORUN W-7: Google Fonts CDN Bagimliligi
  Nerede: src/OzelDers.Web/Components/App.razor ve App/wwwroot/index.html
  Mevcut kod: <link href="https://fonts.googleapis.com/css2?family=Inter...">
  Ne oluyor: Fonts CDN'den geliyor. Yavaş baglantilar veya CDN kesintisinde fontlar yuklenmez,
             tarayici fallback font kullanir (sans-serif). Gorsel tutarsizlik olusur.
             Ayrica her sayfa acilisinda fonts.googleapis.com'a DNS sorgusu gidiyor.
             App projesi icin daha kritik: offline veya zayif ag durumunda uygulama
             yanlis fontlarla acilir.
  Etki: Web - Dusuk | App - Orta/Yuksek

SORUN W-8: Glassmorphism backdrop-filter Performansi
  Nerede: src/OzelDers.SharedUI/wwwroot/css/app.css - .glass ve .glass-dark siniflari
  Mevcut kod: backdrop-filter: blur(12px); -webkit-backdrop-filter: blur(12px);
  Ne oluyor: backdrop-filter: blur() GPU yogun bir islem. Her piksel icin Gaussian blur
             hesaplamasi yapilir. Dusuk GPU'lu cihazlarda (eski Android telefonlar, dusuk
             segmentli Windows laptoplar) bu ciddi FPS dusumune yol acar.
             Arastirma: "CSS blur filter performance - 10-20 FPS drop" (Stack Overflow, 2024).
             Navbar'da, login kartinda, CTA bolumunde kullaniliyor.
             Mobil cihazlarda scroll sirasinda titreme/takilma yasanabilir.
  Etki: Web - Orta | App - Yuksek (dusuk GPU'lu Android cihazlar)

SORUN W-9: touch-action: manipulation Eksik
  Nerede: src/OzelDers.SharedUI/wwwroot/css/reset.css veya utilities.css
  Ne oluyor: Mobil tarayicilarda cift-tiklamayi zoom olarak yorumlayan 300ms gecikme var.
             touch-action: manipulation bu gecikmeyi kaldiriyor.
             Olmadigi durumda her buton tiklamasi 300ms gecikmeli hissettiriyor.
             Bu "native uygulama gibi hissettirmiyor" sikayetinin en buyuk nedenlerinden biri.
  Etki: Web - Orta | App - Yuksek

SORUN W-10: Animasyonlarda will-change Eksik
  Nerede: src/OzelDers.SharedUI/wwwroot/css/utilities.css - .card:hover
  Mevcut kod: .card:hover { transform: translateY(-4px); box-shadow: var(--shadow-hover); }
  Ne oluyor: Hover animasyonu baslamadan once tarayici GPU katmani olusturmuyor.
             will-change: transform eklenirse tarayici onceden GPU katmani hazirlar,
             hover gecisi aninda ve akici olur. Olmadigi durumda ilk hover'da kisa bir
             "jank" (titreme) yasanabilir, ozellikle dusuk GPU'lu cihazlarda.
  Etki: Web - Dusuk/Orta | App - Orta

SORUN W-11: Uzun Listelerde Virtualization Eksik
  Nerede: src/OzelDers.SharedUI/Pages/Search.razor, MyListings.razor
  Mevcut kod: @foreach (var listing in result.Items) { <ListingCard .../> }
  Ne oluyor: Tum ilanlar ayni anda DOM'a render ediliyor. 50-100 ilan oldugunda
             hem ilk render hem de scroll performansi duser.
             Blazor'un built-in <Virtualize> bileseni sadece gorunen elemanlari render eder.
             InteractiveServer modunda bu ozellikle onemli: daha az DOM = daha az SignalR diff.
  Etki: Web - Yuksek (cok ilan oldugunda) | App - Orta

SORUN W-12: Gorsel Lazy Loading Eksik
  Nerede: src/OzelDers.SharedUI/Components/ListingCard.razor
  Mevcut kod: <img src="@Listing.OwnerImageUrl" alt="@Listing.OwnerName" />
  Ne oluyor: Tum ilan fotograflari sayfa acildiginda ayni anda yukleniyor.
             loading="lazy" attribute'u eklenirse tarayici sadece gorunen fotograflari yukler.
             Bu hem ilk sayfa yuklemesini hizlandirir hem de gereksiz bant genisligi tuketimini onler.
  Etki: Web - Orta | App - Orta

BOLUM 6: PAGEHEADER ENTEGRASYONU (Eksik Kalan Kisim)
==================================================

PageHeader.razor bileseni olusturuldu (src/OzelDers.SharedUI/Components/PageHeader.razor)
ama hicbir sayfada kullanilmiyor. Bu bolum hangi sayfalara nasil eklenecegini aciklar.

KURAL: Her iki platformda (Web ve App) UI ayni. PageHeader hem webde hem appte
gozukecek. Fark yok. SharedUI'da yapildiginda otomatik her iki platforma yansir.

PageHeader kullanim ornegi:
  <PageHeader Title="Baslik" Subtitle="Alt baslik" ShowBack="true" BackUrl="/panel/dashboard" />
  veya
  <PageHeader Title="Baslik" ShowBack="true" />  (geri -> history.back())
  veya
  <PageHeader Title="Baslik" ShowBack="false" />  (geri butonu gizli - ana sayfalar)
